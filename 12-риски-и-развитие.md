# 12. Риски и развитие

## Главные риски

| Риск | Вероятность | Влияние | Мера снижения |
|---|---|---|---|
| Ложные слияния идентичности, в т.ч. по семейным/общим билетам | Средняя | Высокое | Два порога + серая зона (clerical review), блокинг, правило «общий контакт ≠ слияние», приоритет якоря, обратимый merge, контроль false-merge rate [ADR-0003] |
| Недоступность якоря (Госуслуги/ЕБС) | Средняя | Среднее | Деградация до контура одного источника, отложенная связка |
| Качество синтетических данных не отражает реальное поведение | Средняя | Высокое | Калибровка по открытым данным мобильности [20, 21], анализ чувствительности |
| Ось мобильности не даёт прироста различимости (H1 не подтверждается) | Низкая | Высокое | Дизайн оценки RFM-only vs RFM+мобильность, честная публикация результата [4, 7] |
| Холодный старт даёт нерелевантный предсегмент | Средняя | Среднее | Пометка `transfer`, быстрый переход в `native`, контроль cold-start lift [12, 16] |
| Пропуск «нужного момента» при сбое | Низкая | Высокое | Durable-таймеры, восстановление из store, окно актуальности [ADR-0008] |
| Двойная доставка или спам уведомлениями | Средняя | Высокое | Дедуп по `dedupe_key`, правило паузы, лимиты, тихие часы [ADR-0009] |
| Экспертные веса скоринга плохо откалиброваны на старте → нерелевантные предложения | Средняя | Среднее | Консервативные пороги `θ`, правило паузы как страховка, калибровка весов по первым откликам, A/B-сравнение версий конфигурации [ADR-0018] |
| Запаздывание/ошибки контекста поездки | Средняя | Среднее | Пересчёт таймеров, отмена снимает intents, алерты точности |
| Рост consumer lag при пике | Средняя | Среднее | Партиционирование по `passenger_id`, автоскейл потребителей |
| Эксплуатационная сложность Kafka | Средняя | Среднее | Управляемый сервис или выделенная команда, мониторинг, бэкап истины в `data-lake` [ADR-0007] |
| Изменение правового контура 152-ФЗ | Низкая | Высокое | Согласия по целям, обезличивание, исполнение ограничения/удаления как код |
| Производные артефакты хранят влияние ПДн после удаления | Средняя | Среднее | Пересчёт признаков/сегментов, переобучение без удалённых данных; machine unlearning как развитие [37] |

## Ограничения MVP

- Сегмент обновляется с каденсом пакетного прогона, а не мгновенно (онлайн-дообучение вне MVP).
- Набор источников, каналов доставки и партнёров минимален и расширяется позже.
- Рекомендация — связка «сегмент → стратегия → действие» с интерпретируемым скорингом `Kop`/`Kpref` [ADR-0018], а не тяжёлая онлайн-модель ранжирования (она отнесена к развитию и сравнивается со скорингом после накопления откликов).
- Пороги правила паузы и точности — стартовые, требуют калибровки на данных апробации.

## Технический долг (осознанный)

- Простой online-ранжировщик внутри `recommendation-service` вместо обучаемой гибридной модели.
- Контекст поездки потребляется по упрощённому контракту; богатая модель этапов — на следующей итерации.
- `trip-context-store` и `notification-store` могут быть консолидированы или, наоборот, разделены при росте нагрузки.

## Решения для пересмотра

| Решение | Когда пересмотреть | Что смотреть |
|---|---|---|
| Брокер Apache Kafka | Узким местом станет онлайн-доставка с микросекундными требованиями | Хвостовая задержка доставки, стоимость эксплуатации, целесообразность гибрида с NATS [ADR-0007] |
| Доставка через каналы потребителей | Появится требование владеть собственным каналом push end-to-end | Надёжность доставки, SLA каналов, дублирование ответственности |
| Пакетная сегментация | Потребуется реакция сегмента в near-real-time | Каденс, дрейф сегментов, нагрузка переобучения |
| Интерпретируемый скоринг `Kop`/`Kpref` | Накоплена история откликов, нужен прирост качества | Объём данных, метрики Precision@k/NDCG, cold-start lift, доля жалоб на навязчивость; эволюция — контекстный бандит с прайором из скоринга при сохранении запретов eligibility [43] [ADR-0018] |
| Границы `trigger`/`notification` | Контур доставки часто меняется отдельно или мешает масштабированию | Частота изменений, инциденты, bottleneck |

## Сигналы, что риск начал реализовываться

- Рост доли спорных слияний или жалоб на «чужие» данные → пересмотр порогов вероятностного связывания.
- Рост доли `expired`/`suppressed` уведомлений или жалоб на навязчивость → калибровка правила паузы и окон.
- Рост consumer lag и хвостовой задержки доставки → масштабирование или пересмотр брокерной схемы.
- Отсутствие статистически значимого прироста различимости сегментов → возврат к дизайну признаков.

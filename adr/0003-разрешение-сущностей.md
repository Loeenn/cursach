# ADR-0003: Разрешение сущностей — детерминированное + вероятностное

## Статус

Принято

## Дата

2026-06-14

## Контекст

Один пассажир распадается на несвязанные записи в разных системах. Чтобы собрать «золотую запись», нужно связать записи. Часть связей очевидна по сильным ключам (подтверждённый аккаунт, телефон + документ в билете), часть — только вероятна (совпадение email + платёжный токен). Слишком жёсткое связывание теряет полноту, слишком вольное — порождает ложные слияния разных людей.

## Решение

Двухступенчатое разрешение сущностей в `identity-service`. Сначала детерминированное связывание по подтверждённым ключам (приоритет якоря Госуслуг/ЕБС, см. [ADR-0004]). Затем вероятностное связывание по модели Феллеги — Сантера [24, 25] для оставшихся записей. Используются **два порога**: выше верхнего — авто-связка, ниже нижнего — разные записи, между — серая зона на ручной разбор (clerical review). Для масштаба применяется **блокинг** (сравнение кандидатов внутри блоков, а не всех пар) [25, 36]. Совпадение общего телефона/email/платёжного токена даёт только кандидата и **не приводит к авто-слиянию разных лиц** (семейные/общие билеты). Все слияния обратимы: история merge хранится, спорные случаи уходят в Steward API для подтверждения или split. Связывание выполняется только в рамках действующего согласия.

## Альтернативы

| Альтернатива | Плюсы | Минусы | Почему не выбрана |
|---|---|---|---|
| Только детерминированное | Минимум ложных слияний | Низкая полнота связывания | Теряем часть «золотой записи» |
| Только вероятностное/ML | Высокая полнота | Ложные слияния, плохая объяснимость | Риск связать разных людей |
| Обучаемая end-to-end модель связывания | Потенциально точнее | Непрозрачна, дорога, требует размеченных данных | Избыточно для MVP, плохо проверяема |
| Детерминированное + вероятностное (Феллеги — Сантер) + Steward | Полнота + контроль + обратимость | Нужны пороги и ручной разбор | Выбрана |

## Последствия

Что стало проще:

- Проверяемость набором тест-кейсов «две записи → один пассажир?».
- Обратимость ошибок через split.

Что стало сложнее:

- Калибровка порогов вероятностной связки.
- Процесс ручного разбора спорных слияний.

Какие риски появились:

- Ложные слияния — снижаются приоритетом якоря, обратимостью и метриками спорных случаев.

## Когда пересмотреть

Если доля спорных слияний станет высокой или появятся качественные размеченные данные для обучаемой модели. Связано с [ADR-0004].

# ADR-0015: Lakehouse для больших данных о поездках и геораспределение по ЦОД

## Статус

Принято

## Дата

2026-06-14

## Контекст

Поведенческие и мобильностные данные поездок — самый объёмный и быстрорастущий поток платформы (порядок ≈ 23 млн пассажиров/год, помноженный на события каждого этапа поездки и смежной мобильности). Эти данные одновременно: (1) источник истины по поведению, (2) обучающая выборка для сегментации и переноса знаний, (3) объект права на удаление по 152-ФЗ. Нужно дешёвое хранение огромных объёмов сырых событий, ACID и управление схемой, воспроизводимость обучения, размещение данных в РФ и устойчивость на уровне ЦОД. «Сырой» data lake не даёт ACID и эволюции схемы; классическое хранилище данных (DWH) дорого для петабайтных append-потоков и плохо хранит сырые события для ML.

## Решение

Реализовать `data-lake` как **собственный lakehouse**:

- **Объектное хранилище** S3-совместимое (self-hosted Ceph/MinIO либо суверенное облако) — дешёвое долговременное хранение; все площадки в РФ (152-ФЗ).
- **Открытый табличный формат Apache Iceberg** поверх объектного хранилища — ACID-таблицы, эволюция схемы и партиций, snapshot/time-travel для воспроизводимого переобучения и аудита [30, 31].
- **Движки** Trino/Spark/Flink, отделённые от хранения; вычисления масштабируются независимо.
- **Партиционирование** по дате события и маршруту/региону; **tiering** горячих и холодных данных по lifecycle-политике.
- **Геораспределение** active-passive: основной ЦОД пишет, резервный получает асинхронную cross-region реплику объектного хранилища; Kafka зеркалируется через cluster linking / MirrorMaker 2; `identity-graph` — primary-write с асинхронной репликой. Горячие витрины региональные и производные — восстанавливаются из lakehouse, а не реплицируются как истина.
- **Удаление** — крипто-шреддингом: ПДн под ключом субъекта (`key-store`, реплицируется в оба ЦОД), удаление ключа стирает данные во всех снапшотах и копиях [ADR-0012].

## Альтернативы

| Альтернатива | Плюсы | Минусы | Почему не выбрана |
|---|---|---|---|
| «Сырой» data lake (только файлы) | Дёшево, просто | Нет ACID, схемы, time-travel; трудная воспроизводимость | Недостаточно для истины и ML |
| Классическое DWH | Удобные запросы, ACID | Дорого на больших объёмах, слабо хранит сырьё для ML | Не подходит для масштаба и переобучения |
| Управляемое зарубежное облако | Меньше эксплуатации | Нарушает резидентность данных РФ (152-ФЗ) | Неприемлемо по закону |
| Один ЦОД + бэкап | Просто | Нет устойчивости к потере ЦОД | Недостаточно для отказоустойчивости |
| Lakehouse (Iceberg) + active-passive мульти-ЦОД | Дёшево, ACID, time-travel, резидентность, DR | Эксплуатация object storage и репликации | Выбрана |

## Последствия

Что стало проще:

- Дешёвое хранение больших данных поездок с ACID и воспроизводимым обучением (time-travel).
- Устойчивость к потере ЦОД и исполнение резидентности данных.

Что стало сложнее:

- Эксплуатация объектного хранилища, Iceberg-каталога и cross-DC репликации.
- Согласованность RPO при асинхронной репликации (резервный ЦОД может отставать).

Какие риски появились:

- Лаг репликации → потеря последних событий при внезапной потере основного ЦОД; снижается частотой репликации и тем, что истина append-only и догоняется.

## Когда пересмотреть

Если потребуется near-real-time доступность во всех ЦОД (active-active), либо объёмы/стоимость изменят выбор формата или хранилища. Связано с [ADR-0001], [ADR-0002], [ADR-0012].

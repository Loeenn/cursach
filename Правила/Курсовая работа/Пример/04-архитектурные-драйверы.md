# 04. Архитектурные драйверы

## Ключевые драйверы

| Драйвер | Почему важен | Влияние на архитектуру | Проверка |
|---|---|---|---|
| Долгая обработка | Синтез книги может занимать минуты или часы | Нужен асинхронный pipeline и worker-процессы | Sequence-диаграммы и тесты очередей |
| Объяснимое списание квоты | Пользователь должен понимать оценку до синтеза | Предобработка завершается до синтеза, manifest хранится отдельно | Проверка quota summary из manifest |
| Надежность задач | Потеря задания неприемлема | MongoDB как источник истины, RabbitMQ только доставляет работу | Тесты повторов и восстановления |
| Большие артефакты | Аудио и исходники нельзя удобно хранить в БД | S3-совместимое хранилище для файлов и архивов | Проверка ссылок на артефакты |
| Telegram как основной UI | Пользователь работает через привычный мессенджер | Нужен отдельный stateless `telegram-bot-adapter` | E2E через Telegram Bot API mock |
| Подписочная монетизация | Доступ к синтезу зависит от активной месячной подписки | Control plane хранит подписку, платежные события и статус тарифа | E2E подписки и webhook tests |
| Недельные квоты | Нужно ограничить дорогой synthesis ресурс | Учет `audio_seconds` по неделям и атомарная проверка перед синтезом | Конкурентные тесты quota reservation |
| Приоритетная обработка | Приоритетный тариф должен получать большую долю capacity | Две synthesis-очереди и weighted scheduler в worker | Тест распределения 70/30 и fallback |
| Лимит выдачи через Telegram | Voice message подходит только для результата до 50 МБ | Нужны два режима доставки: voice и presigned link | Тесты `<= 50 МБ` и `> 50 МБ` |
| Ограниченный срок хранения | Пользовательские аудио не должны храниться бессрочно | Presigned URL на 30 дней и S3 cleanup после 30 дней | Тест cleanup policy |
| Качество речи | TTS чувствителен к пунктуации, языку и длине фрагментов | Отдельные стадии нормализации и chunking | Golden tests для текста |
| Встроенный инференс | ONNX Runtime живет внутри `synthesis-worker` | Нет отдельного inference-сервиса в MVP, но synthesis масштабируется отдельно | ADR и контракт synthesis worker |
| Стоимость обработки | Много маленьких задач и файлов увеличивают накладные расходы | Пакетный синтез и tar-архивы batch result | Проверка batch sizing |

## Главные компромиссы

- Асинхронный pipeline сложнее синхронного API, но лучше подходит для долгой обработки и повторов.
- MongoDB удобна для гибкой модели состояния задания, но требует дисциплины вокруг схемы документов и индексов.
- Хранение batch result в tar-архивах снижает количество объектов, но усложняет частичную диагностику одного фрагмента.
- Разделение workers добавляет инфраструктурную сложность, но упрощает масштабирование разных стадий.
- Telegram ускоряет MVP пользовательского интерфейса, но добавляет зависимость от Bot API и его ограничений.
- Presigned URL на 30 дней удобен пользователю, но требует явной политики cleanup и аккуратной работы с ссылками.
- Подписка проще разовых платежей за каждое задание, но требует надежной обработки рекурентных списаний и отмен.
- Взвешенные очереди проще выделенного пула worker для приоритетного тарифа, но не дают строгой гарантии времени обработки.

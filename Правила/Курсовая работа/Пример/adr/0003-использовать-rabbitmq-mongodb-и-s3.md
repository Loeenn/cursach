# ADR-0003: Использовать RabbitMQ, MongoDB и S3

## Статус

Принято

## Дата

2026-05-12

## Контекст

TTS-система должна хранить состояние заданий, подписок и квот, доставлять фоновые задачи между стадиями и сохранять крупные файлы. Эти задачи имеют разную природу: состояние требует запросов и обновлений, сообщения требуют доставки и повторов, артефакты требуют дешевого долговременного хранения.

## Решение

Использовать три специализированных инфраструктурных компонента:

- RabbitMQ доставляет работу между стадиями pipeline, включая `synthesis.standard` и `synthesis.priority`.
- MongoDB хранит каноническое состояние заданий, пользователей, подписок, платежных событий, недельных квот, прогресса и ссылок на артефакты.
- S3-совместимое object storage хранит исходники, preprocessing manifest, synthesis batch archives и итоговые аудиофайлы.

Правило: RabbitMQ доставляет работу, MongoDB хранит истину, object storage хранит durable artifacts.

## Альтернативы

| Альтернатива | Плюсы | Минусы | Почему не выбрана |
|---|---|---|---|
| Только PostgreSQL | Одна инфраструктурная зависимость | Очереди и большие файлы придется реализовывать поверх БД | Больше самодельной инфраструктуры |
| Kafka вместо RabbitMQ | Хороша для потоков событий и replay | Сложнее для простых work queues MVP | RabbitMQ проще для task pipeline |
| Хранить файлы в MongoDB | Меньше систем | Большие аудиоартефакты неудобны и дороги в БД | Object storage лучше подходит для файлов |
| Redis queue | Быстро и просто | Менее надежная модель долговременных задач без дополнительной дисциплины | Нужна более явная доставка work items |

## Последствия

Что стало проще:

- Разделить состояние, доставку задач и хранение файлов.
- Масштабировать worker-процессы через очереди.
- Хранить большие аудиоартефакты вне базы данных.

Что стало сложнее:

- Нужно эксплуатировать три инфраструктурных компонента.
- Нужно обеспечивать согласованность между MongoDB, RabbitMQ и object storage.
- Нужны правила восстановления при частично выполненной операции.

Какие риски появились:

- Рассинхронизация: файл записан, а ссылка в MongoDB не обновлена.
- Сообщение опубликовано, а состояние задания не соответствует стадии.
- Ошибки прав доступа к object storage.

## Когда пересмотреть

Если команда не сможет надежно эксплуатировать три компонента в учебном MVP, можно временно упростить инфраструктуру. Если появятся требования к audit log и replay событий, стоит рассмотреть event streaming отдельно.

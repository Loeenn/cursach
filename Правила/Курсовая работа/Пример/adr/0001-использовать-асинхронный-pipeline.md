# ADR-0001: Использовать асинхронный pipeline

## Статус

Принято

## Дата

2026-05-12

## Контекст

Синтез длинных книг и статей может занимать минуты или часы. API не должен держать пользовательский HTTP-запрос открытым до завершения обработки. Кроме того, pipeline содержит несколько стадий с разной ресурсной стоимостью и разной вероятностью ошибок: предобработка, синтез и сборка аудио. Синтез также должен учитывать тариф пользователя и недельную квоту.

## Решение

Использовать асинхронный pipeline. Пользователь создает Job через API, получает `job_id`, затем следит за статусом. Долгие стадии выполняются worker-процессами через очередь. Состояние задания хранится отдельно от сообщений очереди. Перед публикацией synthesis-задач control plane проверяет активную подписку, резервирует недельную квоту и выбирает очередь по тарифу.

## Альтернативы

| Альтернатива | Плюсы | Минусы | Почему не выбрана |
|---|---|---|---|
| Синхронная обработка в API | Проще первый прототип | Не подходит для длинных задач, сложнее обрабатывать сбои и таймауты | Противоречит требованиям долгой обработки |
| Один background worker без очереди | Меньше инфраструктуры | Сложнее масштабировать стадии и восстанавливаться после отказов | Плохо подходит для роста нагрузки |
| Готовый workflow engine | Богатая модель workflow | Лишняя сложность для MVP | Можно рассмотреть позже |

## Последствия

Что стало проще:

- Масштабировать тяжелые стадии независимо.
- Повторять отдельные стадии после временных ошибок.
- Показывать пользователю прогресс долгой обработки.

Что стало сложнее:

- Нужно проектировать состояния Job.
- Нужно учитывать повторную доставку сообщений.
- Нужны правила идемпотентности worker-процессов.

Какие риски появились:

- Зависшие задания при рассинхронизации состояния и очереди.
- Дублирующая обработка при повторной доставке сообщений.

## Когда пересмотреть

Если pipeline станет сложным настолько, что ручное управление состояниями и очередями будет часто приводить к ошибкам, стоит рассмотреть workflow engine.

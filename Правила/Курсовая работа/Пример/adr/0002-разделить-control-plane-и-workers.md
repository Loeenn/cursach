# ADR-0002: Разделить control plane и worker-процессы

## Статус

Принято

## Дата

2026-05-12

## Контекст

Система сочетает API-heavy задачи и compute-heavy задачи. API отвечает за пользователей, права доступа, подписки, недельные квоты, состояния, подтверждение оценки и выдачу результата. Worker-процессы выполняют тяжелую обработку текста и аудио. Если смешать эти ответственности, API станет сложнее масштабировать и сложнее защищать от долгих операций.

## Решение

Разделить control plane и worker-процессы:

- `control-plane-api` принимает пользовательские команды, управляет состоянием, подписками, платежными событиями и недельными квотами;
- `telegram-bot-adapter` принимает Telegram updates и преобразует их во внутренние команды;
- `preprocessing-worker` отвечает за извлечение, нормализацию, chunking и оценку расхода квоты;
- `synthesis-worker` отвечает за пакетный синтез и weighted polling тарифных очередей;
- `assembly-worker` отвечает за сборку итогового аудио.

## Альтернативы

| Альтернатива | Плюсы | Минусы | Почему не выбрана |
|---|---|---|---|
| Один backend-сервис со всеми задачами | Быстрее начать разработку | Смешение ответственности, сложнее масштабировать и изолировать сбои | Риск для долгих задач и эксплуатации |
| Выделить каждый шаг в отдельный микросервис сразу | Максимальная независимость | Слишком много инфраструктуры для MVP | Не все границы еще проверены практикой |
| Serverless-функции для каждой стадии | Автомасштабирование | Ограничения времени выполнения и сложности с тяжелым TTS | Не подходит как базовое решение MVP |

## Последствия

Что стало проще:

- Отдельно масштабировать synthesis и assembly.
- Не блокировать API тяжелой работой.
- Изолировать зависимости Telegram Bot API и ONNX Runtime от API.

Что стало сложнее:

- Нужно поддерживать контракты сообщений между стадиями.
- Нужно следить за версионированием manifest и batch result.
- Нужно проектировать диагностику распределенного pipeline.

Какие риски появились:

- Расхождение контрактов между API и worker-процессами.
- Избыточное дробление, если часть workers окажется слишком маленькой.

## Когда пересмотреть

Если в MVP окажется, что отдельная стадия не имеет собственной нагрузки и ответственности, ее можно объединить с соседней стадией. Если наоборот нагрузка резко вырастет, можно дробить worker дальше.

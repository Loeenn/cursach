# ADR

ADR, Architecture Decision Record, это короткая запись об архитектурном решении. ADR нужен, когда решение влияет на структуру системы, данные, инфраструктуру, безопасность, надежность, стоимость или скорость разработки.

## Когда писать ADR

- Выбирается база данных, очередь, хранилище или протокол.
- Меняется граница ответственности компонентов.
- Принимается решение об асинхронной обработке.
- Принимается продуктовое правило с архитектурными последствиями: тарифы, приоритеты, квоты, политика хранения.
- Выбирается архитектурный стиль.
- Есть несколько разумных альтернатив.
- Решение будет сложно поменять позже.

## Именование

Используйте последовательную нумерацию. Название после номера пишите либо на русском кириллицей, либо на английском английскими словами:

```text
0001-использовать-очередь.md
0002-use-message-queue.md
```

Не используйте транслит. Внутри одного проекта лучше выбрать один стиль именования и придерживаться его.

## Статусы

- `Предложено` - решение обсуждается.
- `Принято` - решение действует.
- `Отменено` - решение больше не применяется.
- `Заменено` - решение заменено другим ADR.

## Шаблон ADR

Используйте файл [0001-template.md](0001-template.md) как заготовку для новых ADR.

При создании нового ADR скопируйте файл, замените номер и название, затем заполните разделы под конкретное решение.

## Проверочный список

- ADR короткий и объясняет конкретное решение.
- Есть контекст проблемы.
- Есть альтернативы.
- Есть последствия и риски.
- Понятно, когда решение стоит пересмотреть.
- ADR не пересказывает весь раздел архитектуры, а фиксирует одно решение и его компромиссы.
